## 核心规则

design.md 只回答一个问题：**怎么做**。
它承接 spec.md（做什么），输出给 plan.md（怎么拆）。

如果你在写用户故事、验收标准、产品约束 → 停下，那属于 spec.md。
如果你在写任务列表、执行顺序、agent 分配 → 停下，那属于 plan.md。

## 边界判断表

| 放 spec.md | 放 design.md | 放 plan.md |
|------------|-------------|------------|
| 做什么、为什么 | 怎么做 | 怎么拆、谁执行 |
| 用户场景、验收标准 | 数据模型、接口契约、核心流程 | 任务列表、依赖关系、验证命令 |
| 产品约束 | 技术约束、性能约束 | 风险、阻塞、完成标准 |
| 功能边界 | 影响范围（改哪些模块） | 执行模式（单 agent / subagent） |
| 异常的用户体验 | 异常的技术处理策略 | — |

## 写作步骤（按此顺序执行）

### Step 1: 背景
- 从 spec.md 概括动机，一段话，不要复制
- 如果 spec.md 还没写完 → 先回去补 spec

### Step 2: 技术方案
分三个子节，缺一不可：

**数据模型**：新增/修改的实体、字段、关系。用表格或代码块。
**接口契约**：从 spec 的"输入与输出"映射为技术接口（API 端点 / 函数签名 / 消息格式）。
**核心流程**：用 Mermaid 画主流程，对应 spec 中的用户场景。

### Step 3: 影响范围
- 列出需要修改的模块/文件 + 变更类型（新增/修改/删除）
- **这张表直接决定 plan.md 的任务拆分**——如果这里不清楚，plan 就拆不准

### Step 4: 约束
- 从 spec 的"产品约束"继承
- 叠加技术约束（向后兼容、性能指标等）
- 每条约束必须可验证——如果不能写成测试或检查命令，重写

### Step 5: 迁移与兼容（涉及 schema/API/数据格式变更时必填）
- Schema migration：迁移脚本路径或 DDL
- 数据回填：是否需要、策略、预估耗时
- 向后兼容：旧客户端/旧数据如何处理
- Feature flag：flag 名称和启用条件
- 不涉及变更 → 写"不适用"，不要留空

### Step 6: 发布与回滚
- 发布策略（全量/灰度/分批）
- 回滚方案 + 回滚后数据如何处理
- 回滚触发条件（什么指标异常时触发）

### Step 7: 观测性
- 需要新增的 metrics / 告警规则 / 结构化日志或 span

### Step 8: 异常处理
- 从 spec 的"异常与边界情况"映射为技术处理策略（重试/降级/报错/熔断）

### Step 9: 验证方式
- 从 spec 的验收标准映射为测试策略
- 必须覆盖：单元测试（核心逻辑）+ 集成测试（端到端流程）
- 有性能约束 → 加性能测试

### Step 10: 备选方案
- 至少评估一个替代方案
- 每个方案写：优势 + 否决原因
- 如果没有备选 → 你没充分思考，至少考虑"不做"这个选项

## 自检清单（写完后逐条检查）

- [ ] 没有产品层面内容（对照边界判断表）
- [ ] 没有任务拆分内容（那属于 plan.md）
- [ ] 数据模型、接口契约、核心流程三个子节都已填写
- [ ] 影响范围表已列出所有需要修改的模块
- [ ] 每条约束可验证
- [ ] 迁移与兼容已填写（或明确标注"不适用"）
- [ ] 至少一个备选方案
- [ ] frontmatter 的 id/status/owner/created/verified 已填写
