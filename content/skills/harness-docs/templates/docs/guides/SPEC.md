## 核心规则

spec 只回答两个问题：**做什么** 和 **为什么**。
如果你在写状态码、字段类型、数据库表结构、缓存策略 → 停下，移到 design.md。

## 边界判断表

写内容时逐条对照。左列放 spec，右列放 design.md。

| 放 spec.md | 放 design.md |
|------------|-------------|
| 用户能感知的输入输出 | API 端点、HTTP 状态码、字段类型 |
| 用户感知的响应速度（"3 秒内"） | 缓存策略、数据库索引、p99 指标 |
| 业务规则（"密码至少 8 位"） | 实现规则的技术方案 |
| 错误场景的用户体验 | 错误处理策略（重试/熔断/降级） |
| 产品约束（"不改现有工作流"） | 技术约束（"向后兼容 API v2"） |

## 写作步骤（按此顺序执行）

### Step 1: 问题与动机
- 写清用户遇到什么问题、当前方案的痛点
- 如果说不清为什么做 → 暂停，先和人类确认

### Step 2: 功能边界
- In Scope：列出本次要实现的能力（每条一行）
- Out of Scope：列出明确排除的内容
- **Out of Scope 比 In Scope 更重要**——它防止 design 阶段范围蔓延

### Step 3: 用户场景
- 每个场景用编号步骤：用户做什么 → 系统响应 → 用户看到什么
- 复杂流程用 Mermaid
- 每个场景必须独立成段，有标题

### Step 4: 输入与输出
- 只写用户视角：用户提供什么、看到什么
- 约束只写用户可感知的（如"密码至少 8 位"），不写技术实现

### Step 5: 验收标准
- 从 Step 3 的每个场景推导，每个场景至少一条
- 格式：Given [前置条件], When [操作], Then [预期结果]
- 每条必须可测试——如果不能写成自动化测试，重写

### Step 6: 异常与边界
- 从 Step 5 的反面推导：正常流的每个步骤，如果失败会怎样？
- 必须填触发条件 + 预期行为

### Step 7: 产品约束 + 度量
- 约束：不可违反的产品层面限制
- 度量：上线后怎么判断成功

## 自检清单（写完后逐条检查）

- [ ] 没有技术实现细节（对照边界判断表）
- [ ] Out of Scope 已明确列出
- [ ] 每个用户场景都有对应的验收标准
- [ ] 每条验收标准可写成自动化测试
- [ ] 异常场景覆盖了正常流每个步骤的失败情况
- [ ] frontmatter 的 id/status/owner/created/updated 已填写

## 拆分规则

一个 spec 的功能边界应该是一个可独立交付的单元。判断标准：
- 能独立上线、独立回滚 → 粒度合适
- 必须和另一个需求同时上线才有意义 → 合并为一个 spec
- In Scope 超过 10 条 → 考虑拆分
