# 文档审查方法论

本项目采用三轮自审查机制，确保 spec.md、design.md、plan.md 的质量。每轮关注不同维度，每个维度 1-5 分，通过线 16/20 分，最多 2 次修复机会。

## 审查流程

```
写文档 → R1 结构审查 → R2 逻辑审查 → R3 可执行性审查 → 通过 / 修复 / 拒绝
```

每轮审查 4 个维度，每维度 1-5 分：
- **5 分**：完全符合标准，无需改进
- **4 分**：基本符合，有小瑕疵但不影响理解
- **3 分**：部分符合，有明显问题需修复
- **2 分**：大部分不符合，需要重写
- **1 分**：完全不符合标准

**通过标准**：总分 ≥ 16/20，且无 1 分项。

**修复机制**：
- 首次未通过：指出问题，给出修复建议，允许修复后重审
- 二次未通过：再次修复，最后一次重审
- 三次未通过：拒绝文档，需重新设计方案

## R1 — 结构审查

检查文档是否符合模板结构、必填字段完整、章节逻辑清晰。

### spec.md 结构维度

1. **Frontmatter 完整性** (1-5)
   - 5: slug/status/updated 齐全且格式正确
   - 3: 缺少 1 个字段或格式错误
   - 1: 缺少 2+ 字段

2. **Given/When/Then 完整性** (1-5)
   - 5: 所有场景都有完整的 G/W/T
   - 3: 部分场景缺少 When 或 Then
   - 1: 大部分场景只有 Given

3. **Out-of-Scope 明确性** (1-5)
   - 5: 明确列出不做什么，有理由
   - 3: 列出了但没有理由
   - 1: 没有 Out-of-Scope 或只说"暂不考虑"

4. **章节逻辑** (1-5)
   - 5: 问题→方案→验收→边界，逻辑清晰
   - 3: 章节顺序混乱或有跳跃
   - 1: 缺少关键章节

### design.md 结构维度

1. **Frontmatter 完整性** (1-5)
   - 同 spec.md

2. **三技术子章节完整性** (1-5)
   - 5: 数据模型、接口契约、依赖关系三者齐全
   - 3: 缺少 1 个子章节
   - 1: 缺少 2+ 子章节

3. **约束可验证性** (1-5)
   - 5: 每个约束都有验证方式（测试/命令/工具）
   - 3: 部分约束有验证方式
   - 1: 约束都是描述性的，无验证方式

4. **章节逻辑** (1-5)
   - 5: 方案→技术细节→约束→风险，逻辑清晰
   - 3: 章节顺序混乱
   - 1: 缺少关键章节

### plan.md 结构维度

1. **Frontmatter 完整性** (1-5)
   - 同 spec.md

2. **任务字段完整性** (1-5)
   - 5: 所有任务都有 description/module/verify/estimate
   - 3: 部分任务缺少 verify 或 estimate
   - 1: 大部分任务只有 description

3. **Verify 命令可执行性** (1-5)
   - 5: 所有 verify 都是可执行命令
   - 3: 部分 verify 是描述性的
   - 1: 大部分 verify 不可执行

4. **章节逻辑** (1-5)
   - 5: 任务→依赖→决策→风险，逻辑清晰
   - 3: 章节顺序混乱
   - 1: 缺少关键章节

## R2 — 逻辑审查

检查文档内容的逻辑一致性、完整性、可行性。

### spec.md 逻辑维度

1. **问题定义清晰度** (1-5)
   - 5: 问题有具体场景、数据、影响范围
   - 3: 问题描述模糊，缺少具体场景
   - 1: 问题定义不清或与方案不匹配

2. **验收标准可测试性** (1-5)
   - 5: 每个 Then 都可以写成测试用例
   - 3: 部分 Then 是描述性的
   - 1: Then 都是主观判断

3. **边界合理性** (1-5)
   - 5: Out-of-Scope 与 In-Scope 互补，无遗漏
   - 3: 有灰色地带或重叠
   - 1: 边界不清或矛盾

4. **方案完整性** (1-5)
   - 5: 方案覆盖所有 Given/When/Then
   - 3: 部分场景没有对应方案
   - 1: 方案与验收标准脱节

### design.md 逻辑维度

1. **方案可行性** (1-5)
   - 5: 方案有技术依据，风险可控
   - 3: 方案有风险但未评估
   - 1: 方案不可行或缺少关键细节

2. **接口契约完整性** (1-5)
   - 5: 所有对外接口都有输入/输出/错误定义
   - 3: 部分接口缺少错误处理
   - 1: 接口定义不完整

3. **依赖关系正确性** (1-5)
   - 5: 依赖方向符合 ARCHITECTURE.md
   - 3: 有循环依赖或违反分层
   - 1: 依赖关系混乱

4. **约束覆盖度** (1-5)
   - 5: 约束覆盖性能/安全/可靠性
   - 3: 只有功能约束
   - 1: 没有约束或约束不合理

### plan.md 逻辑维度

1. **任务拆分合理性** (1-5)
   - 5: 任务粒度适中，依赖清晰
   - 3: 部分任务过大或依赖不清
   - 1: 任务拆分不合理

2. **依赖关系正确性** (1-5)
   - 5: 任务依赖形成 DAG，无循环
   - 3: 有隐含依赖未声明
   - 1: 依赖关系错误或循环

3. **估算合理性** (1-5)
   - 5: 估算有依据，考虑了风险
   - 3: 估算过于乐观
   - 1: 估算明显不合理

4. **决策记录完整性** (1-5)
   - 5: 关键决策都有理由和权衡
   - 3: 部分决策缺少理由
   - 1: 没有决策记录

## R3 — 可执行性审查

检查文档是否可以直接指导实施，验证命令是否可执行。

### spec.md 可执行性维度

1. **验收标准可测试性** (1-5)
   - 5: 每个 Then 都可以写成自动化测试
   - 3: 部分 Then 需要人工验证
   - 1: Then 无法验证

2. **场景覆盖度** (1-5)
   - 5: 覆盖正常/异常/边界场景
   - 3: 只覆盖正常场景
   - 1: 场景不完整

3. **数据具体性** (1-5)
   - 5: 场景有具体数据示例
   - 3: 场景是抽象描述
   - 1: 场景缺少数据

4. **可追溯性** (1-5)
   - 5: 每个需求都可以追溯到代码/测试
   - 3: 部分需求难以追溯
   - 1: 需求与实现脱节

### design.md 可执行性维度

1. **接口可实现性** (1-5)
   - 5: 接口定义可以直接写代码
   - 3: 接口定义需要补充细节
   - 1: 接口定义不完整

2. **约束可验证性** (1-5)
   - 5: 每个约束都有验证命令
   - 3: 部分约束需要人工验证
   - 1: 约束无法验证

3. **数据模型可实现性** (1-5)
   - 5: 数据模型可以直接写 schema
   - 3: 数据模型需要补充字段
   - 1: 数据模型不完整

4. **风险缓解措施具体性** (1-5)
   - 5: 每个风险都有具体缓解措施
   - 3: 部分风险只有描述
   - 1: 风险没有缓解措施

### plan.md 可执行性维度

1. **Verify 命令可执行性** (1-5)
   - 5: 所有 verify 都是可执行命令
   - 3: 部分 verify 需要人工验证
   - 1: verify 不可执行

2. **任务可分配性** (1-5)
   - 5: 任务描述清晰，可以直接分配
   - 3: 部分任务需要补充细节
   - 1: 任务描述不清

3. **进度可追踪性** (1-5)
   - 5: 任务状态可以自动更新
   - 3: 任务状态需要人工更新
   - 1: 任务状态难以追踪

4. **回滚可操作性** (1-5)
   - 5: 回滚步骤清晰，可以直接执行
   - 3: 回滚步骤需要补充细节
   - 1: 没有回滚方案

## 审查示例

### spec.md 审查示例

**R1 结构审查**：
- Frontmatter: 5 分（齐全）
- G/W/T: 4 分（部分场景缺少 When）
- Out-of-Scope: 5 分（明确且有理由）
- 章节逻辑: 5 分（清晰）
- **总分**: 19/20，通过

**R2 逻辑审查**：
- 问题定义: 5 分（清晰）
- 验收标准: 3 分（部分 Then 是描述性的）
- 边界合理性: 5 分（合理）
- 方案完整性: 4 分（基本覆盖）
- **总分**: 17/20，通过

**R3 可执行性审查**：
- 验收标准: 3 分（部分需要人工验证）
- 场景覆盖: 4 分（缺少边界场景）
- 数据具体性: 5 分（有示例）
- 可追溯性: 5 分（可追溯）
- **总分**: 17/20，通过

**最终结果**：三轮都通过，文档质量合格。

## 适用范围

本审查方法论适用于：
- 中任务：design.md + plan.md
- 大任务：spec.md + design.md + plan.md

小任务无需正式文档，不适用本方法论。

## 审查工具

- 人工审查：按本文档标准逐项打分
- 自动化审查：`lint-docs.ts` 检查结构完整性
- CI 集成：在 PR 中自动运行 lint-docs.ts
